# AI Write Workspace Recovery And Style DNA Design

日期：2026-04-17

## 背景

当前 `ai-write` 有两条需要并行治理但应该分阶段落地的线：

1. 工作区主链路失真
   - 分卷大纲生成后刷新丢失
   - 自动创建了卷，但卷内没有稳定可见的大纲和章节结构
   - PC 端缺失整体大纲查看/编辑环节
   - 项目、卷、章缺少删除入口
2. 风格检测提示词偏泛，难以提炼“像这个人写的”可执行风格 DNA

这两条线共享同一个原则：先让数据边界和落点稳定，再增强提示词质量。

## 子项目拆分

### 子项目 A：工作区恢复与持久化收口

目标是让书 / 卷 / 章大纲与卷章实体形成一个刷新后可恢复、PC/手机一致的持久化工作流。

### 子项目 B：风格 DNA 提取优化

目标是让风格检测从“泛分析”升级为“可迁移、可复用、可追溯”的写法档案生成。

## 推荐顺序

推荐先做 A，再做 B。

原因：

- A 是主流程故障，直接影响能否使用系统
- B 属于能力增强，依赖的 `StyleProfile` 主结构已经存在，不阻塞主流程修复
- 两者都要动后端服务与前端展示，但 A 更高优先级、更高风险

## 方案对比

### 方案 1：只修前端显示

做法：

- 在 `DesktopWorkspace.tsx` 里补更多本地状态恢复
- 把已有 outline 文本重新塞回 PC 视图

优点：

- 见效快
- 改动面小

缺点：

- 根因没有解决
- outline 自动保存和手动保存仍然是双轨
- 刷新后状态和数据库状态仍然可能不一致

结论：不推荐。

### 方案 2：收口持久化边界，前端全部从数据库重建工作区

做法：

- 明确谁负责创建 outline
- 明确卷和章如何从 persisted outline 派生
- PC 与手机统一走“加载项目快照并还原界面”逻辑

优点：

- 根因级修复
- 刷新一致性最好
- 后续补编辑/删除能力更顺手

缺点：

- 需要同时动前后端
- 需要补测试和加载模型

结论：推荐。

### 方案 3：直接重做工作区

做法：

- 重写 PC/手机工作区和大纲向导

优点：

- 可以一次性把 UX 做完整

缺点：

- 风险高
- 周期长
- 不符合当前仓库已有结构

结论：不推荐。

## 设计方案

### A. 工作区恢复与持久化收口

#### 1. 根因判断

当前问题不是“生成失败”，而是“生成结果没有被桌面工作区稳定消费”。

具体表现：

- `DesktopWorkspace.tsx` 用本地 `outlinePreview` 和 `wizardStep` 驱动流程
- 刷新后，本地状态丢失，只重新加载部分持久化数据
- `generate.py` 自动保存 outline，前端又手动 POST outline，导致单一事实来源不存在
- 章节生成时主要靠从文本里抽标题再创建记录，而不是从稳定结构化 outline 派生

#### 2. 目标状态

一个项目在任意时刻都应该能从数据库恢复出以下快照：

- 当前书级大纲
- 已确认的卷级大纲
- 每卷对应的卷记录
- 每卷对应的章节记录
- 每章已保存的章级大纲

前端页面刷新后，不依赖内存态也能恢复：

- 现在处于哪一步
- 已经生成到哪一层
- 用户当前还能做哪些动作

#### 3. 数据边界

建议把以下边界固定下来：

- `generate.py` 负责生成，不负责偷偷落库重复数据
- 显式保存由业务接口统一完成
- `Outline` 表负责存书 / 卷 / 章三级大纲
- `Volume` / `Chapter` 表负责可编辑实体
- `Chapter.outline_json` 只保存该章可直接用于正文生成的结构化章纲

这样可以保证：

- outline 层级用于规划
- volume/chapter 实体用于操作
- 两者不是互相替代，而是有明确映射

#### 4. 前端改造原则

PC 端应补齐一个“项目快照视图模型”，至少要能从持久化数据推断：

- 没有书纲：显示步骤 1
- 有书纲没卷：显示步骤 2，并能查看/编辑书纲
- 有卷没章：显示步骤 3，并能查看卷级结构
- 有章：进入编辑工作区，同时允许回看大纲

同时补齐操作入口：

- 删除项目
- 删除卷
- 删除章节
- 查看整体大纲
- 编辑整体大纲

第一阶段不追求完美重设计，优先把已有页面补成“可恢复、可操作、可删除”。

#### 5. 文件边界建议

建议把 `DesktopWorkspace.tsx` 的职责切开：

- 页面容器继续留在 `DesktopWorkspace.tsx`
- 项目快照加载与阶段推导提取成独立 helper
- 大纲向导提取成独立组件
- 项目管理动作提取成独立组件

这样后续继续补“编辑大纲”“重生成某一卷/某一章”时不会再把主文件堆得更大。

### B. 风格 DNA 提取优化

#### 1. 当前架构约束

当前 `StyleProfile` 结构已经足够承载第一阶段增强，不建议一开始做数据库迁移。

现有字段可以这样复用：

- `rules_json`：放可执行写作规则
- `anti_ai_rules`：放明确的负面表达约束
- `tone_keywords`：放风格标签/关键词
- `sample_passages`：放代表性段落
- `config_json`：放完整的风格 DNA 原始报告和逐字摘录证据

#### 2. 推荐接法

你提供的提示词不适合直接拿来当“最终写作 prompt”，但非常适合拿来做“风格抽取 prompt”。

推荐做法：

- 把它接入 `style_detection.py` 的 LLM 深度分析阶段
- 输出结果进入 `config_json.style_dna_report`
- 再由一个映射层，把报告压缩进 `rules_json` / `tone_keywords` / `sample_passages`

这样有两个好处：

- 对现有生成链路侵入小
- 后续想在前端展示更完整的分析结果时，不需要重跑检测

#### 3. 提示词层面的必要收束

原始版本信息量很强，但直接上会有三个风险：

1. 输出太散，不稳定
2. “逐字提取”要求容易被模型违反
3. 最后的结果很难无损落进当前 `StyleProfile`

所以要做三层收束：

- 明确只输出 JSON，不输出 Markdown
- 明确“逐字摘录”字段和“归纳字段”分开
- 明确每个字段的长度与数量上限，避免结果爆炸

#### 4. 目标输出结构

第一阶段建议让风格 DNA 返回以下结构化字段：

- `persona_summary`
- `language_features`
- `verbatim_phrase_groups`
- `writing_patterns`
- `forbidden_patterns`
- `representative_quotes`
- `style_labels`
- `distilled_rules`

其中：

- `verbatim_phrase_groups` 和 `representative_quotes` 必须逐字摘录
- `distilled_rules` 用于写作时直接注入
- `forbidden_patterns` 可转成负向规则

#### 5. 与现有生成链路的结合方式

第一阶段不改前端交互，只增强后台生成质量：

- `styles.py /detect` 继续保持原入口
- 后端检测完成后生成更高质量的 `StyleProfile`
- `compile-preview`、`test-write`、章节生成会自然吃到更强的规则集

第二阶段如果需要，再在 `styles/page.tsx` 增加“查看风格 DNA 报告”的展开面板。

## 非目标

这轮不做：

- 全量重写手机工作区
- 大规模数据库 schema 重构
- 直接把风格 DNA 报告完整展示到前端
- 把写法系统改成完全不同的新模型

## 验证策略

### 工作区

- 创建项目后生成全书大纲、分卷、章节
- 刷新页面后确认 PC/手机都能恢复
- 删除项目 / 卷 / 章后界面与数据库同步更新
- 已有项目能正常加载，不因新逻辑丢失旧数据

### 风格 DNA

- 从文本检测写法时，能稳定返回结构化报告
- `rules_json` 不为空，且比当前版本更可执行
- `config_json` 中保留完整风格 DNA 报告
- 试写结果能体现更明确的语言习惯和禁忌

## 下一步

下一步不应该写一个混合大计划，而应该拆成两个实施计划：

1. `工作区恢复与 CRUD 补齐`
2. `风格 DNA 抽取接入与映射`

两个计划可以共享同一份架构文档，但执行上分开推进。
