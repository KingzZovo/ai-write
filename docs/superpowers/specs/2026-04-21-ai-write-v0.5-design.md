# AI Write v0.5 设计规格

**日期：** 2026-04-21
**状态：** Draft（待实施计划）
**上一版本：** v0.4.0（2026-04-19）

## 1. 背景与目标

v0.4 交付后暴露了三个系统性问题：

1. **任务路由僵化**：`ModelConfig` 表硬编码 7 个 `task_type`（`outline / generation / polishing / extraction / summary / evaluation / embedding`），但 `PromptRegistry` 内建已有 9 个 prompt 且可任意扩展；新建的 prompt 无法独立分配端点或覆盖模型参数。
2. **向量化不透明**：Qdrant 的 `plots / styles / chapter_summaries` 三个 collection 在知识库导入和章节摘要时被写入，但前端没有任何查看、搜索、删除或重建的入口；用户无法判断"提取得合不合理"。
3. **RAG 实际未启用**：`api/generate.py` → `ChapterGenerator` → `context_assembler.build_context_for_chapter`（老路径），完全绕过了 `ContextPackBuilder._build_rag()`；`chapter_summaries` 写入了却从不被检索，生成时没有 RAG。

配套目标：
- 可以看到每一次 LLM 调用的完整对话（messages + RAG 命中 + 响应 + tokens），否则无法验证上述修复是否真的生效。
- 把散落的"能力"结构化，给未来的 capability toggle / agent 化留口子（参考 NovelClaw 的 `capability_registry`）。
- 长生成过程中允许 LLM 反问作者（参考 NovelClaw 的 `ask_user` 文件 IPC）。

v0.5 作为一次整合发布，一次性解决这五件事。

## 2. 非目标

- 不做完整 ReAct agent 循环（NovelClaw 的 `claw_manager` 式自主 tool calling）——风险高、改动大，留 v0.7+。
- 不做 NovelClaw 的 16 Claw Memory Banks 分类——需要重构 `Character.profile_json / WorldRule`，留 v0.6。
- 不做 reward-based 迭代重写循环——依赖 6 Checker 评分标准化，留 v0.6。
- 不做 NovelClaw 的转折点生成时检测——与现有 CFPG 伏笔系统重叠，需要先对齐语义，留 v0.6。
- 不引入新 LLM provider（保持 `anthropic / openai / openai_compatible` 三类）。

## 3. 核心设计决策

| 决策 | 选择 | 备选被拒原因 |
|-----|------|-----------|
| 路由粒度 | **每个 Prompt 直绑 endpoint/model/temperature/max_tokens** | "保留 7 固定 task_type + Prompt override" 造成双层语义混乱；"用户自定义 task_type 列表"仍需中间概念，不如直接 Prompt |
| 向量化面板权限 | **可读写 + 重建索引** | 只读不够（embed 模型换代后无法清理）；带历史检索日志过度 |
| 对话观测 | **持久化 `LLMCallLog` 表 + `/logs` 页** | 只用 logger.info 不便非技术用户；仅当次预览失去历史对照价值 |
| RAG 冷启动 | **一键回溯生成** | 仅对新生成生效会让老项目长期无 RAG；后台自动补差异增加复杂度 |
| 发布节奏 | **v0.5 单次大升级** | 多个小版本增加迁移脚本数量，收益不大 |
| NovelClaw 借鉴范围 | **仅 capability 字段 + ask_user IPC** | memory banks / reward loop / turning point 都需要先打地基，v0.5 塞不下 |

## 4. 架构总览

```
              ┌──────────── /prompts (扩) ────────────┐
              │   PromptAsset: + endpoint_id +        │
              │   model_name + temperature + max_tokens│
              │   + category + order + always_enabled │
              │                                       │
              │        ↓ resolve_route(task_type)     │
              │   PromptRegistry (唯一路由入口)       │
              └──────────┬────────────────────────────┘
                         │
   ┌─────────────────────┼─────────────────────┐
   ↓                     ↓                     ↓
ChapterGenerator   PlotExtractor         SummaryAgent / Evaluator
  │                  │                      │
  ↓                  ↓                      ↓
ContextPackBuilder  直接 run_*_prompt     直接 run_*_prompt
  │ (L1+L2+L3 RAG 命中)
  ↓
ModelRouter.stream(endpoint_id, model)
  │
  ↓  (llm_call_logger 装饰器)
LLMCallLog (messages, rag_hits, tokens, latency)
  │
  ↓
/logs 页面  (按 project/chapter 过滤)

并行：
  /vector 面板 → QdrantStore CRUD · 检索测试 · rebuild-rag
  workspace → SSE 订阅 ask_user → AskUserPrompt 弹窗
```

核心变更：
1. `ModelConfig`（7 固定 task 路由表）→ **删除**；所有路由下沉到 `PromptAsset` 新字段。
2. `ChapterGenerator` **重写**：直接调 `ContextPackBuilder`，删除 `context_assembler.py`。
3. 新表 `LLMCallLog` + 新 API `/api/call-logs`；`run_*_prompt` 与 `stream_*_prompt` 包装在 `llm_call_logger` 里自动写日志。
4. 新 API `/api/vector-store/*` + 新页 `/vector`（统计、payload 预览、检索测试、删点、重建）。
5. 新 API `/api/projects/{id}/rebuild-rag`（Celery 任务，扫全部章节补 summary + embed）。
6. `PromptAsset` 加 `category / order / always_enabled / name_en / description_en`，`/prompts` 页按 category 分组。
7. 新表 `AskUserPause` + 新 API `/api/ask-user/*` + workspace 弹窗组件。

## 5. 组件清单

### 5.1 数据模型

#### `PromptAsset`（扩字段）

新增列：

```python
endpoint_id: UUID | None         # FK → LLMEndpoint.id, ON DELETE SET NULL
model_name: str | None           # 覆盖 endpoint.default_model
temperature: float = 0.7
max_tokens: int = 4096
category: str = "Core"           # 如 "Core Writing" / "Extraction" / "Evaluation"
order: int = 0                   # 前端排序
always_enabled: int = 0          # 1 = 不可在 UI 上禁用
name_en: str | None              # 中英双语
description_en: str | None
```

`task_type` 保留为字符串（不是枚举），允许未来任意扩展。

#### `ModelConfig`（删除）

Alembic 迁移：把表中每条 `(task_type, endpoint_id, model_name, temperature, max_tokens)` 映射到对应 active `PromptAsset`（按 `task_type` 匹配），写到新字段后 `op.drop_table("model_config")`。匹配不上的 task_type 按 `BUILTIN_PROMPTS` 补齐。

#### `LLMCallLog`（新建）

```python
id: UUID (pk)
prompt_id: UUID | None       # FK → PromptAsset.id, SET NULL
task_type: str               # 冗余存储，便于没 prompt 时也能过滤
project_id: UUID | None
chapter_id: UUID | None
messages_json: JSONB         # 完整发出去的 messages
rag_hits_json: JSONB | None  # [{collection, score, payload}]
response_text: text
input_tokens: int = 0
output_tokens: int = 0
latency_ms: int = 0
model: str                   # 实际用的 model name
endpoint_id: UUID | None
status: str                  # 'ok' / 'error'
error_message: str | None
created_at: datetime (idx)
```

索引：`(project_id, created_at desc)`, `(chapter_id, created_at desc)`, `(task_type)`.

保留策略：无自动清理；前端 `/logs` 提供"按项目清理"按钮。

#### `AskUserPause`（新建）

```python
id: UUID (pk)
project_id: UUID
chapter_id: UUID | None
run_id: UUID                 # 同一次生成任务的所有 pause 共享
question: text
answer: text | None
status: str                  # 'pending' / 'answered' / 'timeout' / 'cancelled'
created_at: datetime
answered_at: datetime | None
timeout_at: datetime         # created_at + timeout (default 5min)
```

索引：`(project_id, status, created_at desc)`.

| 文件 | 变更 |
|------|------|

| `services/chapter_generator.py` | **重写**。`generate` / `generate_stream` 接收 `project_id, volume_id, chapter_idx, user_instruction`；内部调 `ContextPackBuilder.build(...)` → `pack.to_messages(user_instruction)` → `stream_text_prompt("generation", messages=...)` |

| 路径 | 变更 |
|------|------|

## 6. 数据流

### 6.1 章节正文生成（RAG 真正生效）

```
POST /api/generate/chapter (SSE)
  │
  ├─ ContextPackBuilder.build(project_id, volume_id, chapter_idx)
  │    ├─ L1: 最近 5 章摘要 + 本章已有内容 + 当前大纲 + 后 10 章走向
  │    ├─ L2: world_rules + character_cards(SCORE) + CFPG + timeline + strand
  │    └─ L3: _extract_entities_from_outline → generate_embedding
  │           → qdrant.search("chapter_summaries", filter={project_id})
  │           → pack.rag_snippets (top_k=5, score≥0.4)
  │           → dialogue_samples, style_samples
  │
  ├─ PromptRegistry.resolve_route("generation") → RouteSpec
  │
  ├─ messages = pack.to_messages(user_instruction)
  │
  ├─ async with llm_call_logger(task_type="generation", prompt_id, project_id,
  │                              chapter_id, rag_hits=pack.rag_snippets):
  │      ModelRouter.stream_by_route(route, messages)
  │        ├─ 前端 SSE: data: {chunk: "..."}
  │        └─ 累计 response_text, usage
  │
  ├─ finalize:
  │    ├─ Chapter.content_text = response_text
  │    ├─ 异步 Celery: summary + embed → chapter_summaries
  │    └─ LLMCallLog 由 logger 上下文管理器在 exit 时写入
  │
  └─ 前端 SSE: data: {event: "done", log_id}
```

### 6.2 ask_user 打断

```
Agent 调 AskUserService.ask("主角是否死亡？")
  │
  ├─ INSERT AskUserPause(question, status=pending, timeout=+5min) → id
  ├─ Redis PUB "ask_user:{project_id}" → {id, question}
  │
  ├─ [前端 workspace] SSE 订阅 "ask_user:{project_id}"
  │    ├─ 弹 AskUserPrompt 对话框
  │    └─ 用户答 → POST /api/ask-user/{id}/answer
  │         ├─ UPDATE status=answered, answer=...
  │         └─ Redis PUB "ask_user:answer:{id}"
  │
  └─ [后端 Service] 阻塞等 "ask_user:answer:{id}" 或 timeout
       ↓ 收到 → 返回 answer
       ↓ 超时 → UPDATE status=timeout → 返回 "(no reply — timed out)"
```

### 6.3 Qdrant 回溯重建

```
POST /api/projects/{id}/rebuild-rag { force: false }
  │
  ├─ 启动 Celery task rebuild_rag_for_project
  │    ├─ 扫 Chapter WHERE content_text != "" AND volume.project_id = id
  │    ├─ 按 chapter_idx 升序遍历：
  │    │   ├─ 若 qdrant 已有 point_id=hash(volume_id+chapter_idx) 且 !force → skip
  │    │   ├─ run_structured_prompt("summary", content_text) → summary
  │    │   ├─ generate_embedding(summary) → vector
  │    │   └─ qdrant.upsert("chapter_summaries", point)
  │    ├─ 每 5 章写 Redis 进度：{done, total, current_chapter, failed: []}
  │    └─ 收尾：{done, total, status: "completed"/"partial", failed}
  │
  └─ 前端 /vector 轮询 GET /api/vector-store/rebuild-progress?project_id=X
```

## 7. 错误处理

| 情况 | 策略 |
|------|-----|
| PromptAsset 没绑 endpoint | `resolve_route` 抛 `ValueError("Prompt X 未配置端点")`；前端生成面板显示并跳转 `/prompts` |
| endpoint 被删但 Prompt 还指向它 | 外键 `ON DELETE SET NULL`；`resolve_route` 回退到第一个可用 endpoint + `logger.warning` |
| `LLMCallLog` 写入失败 | `logger.warning`，**不阻塞生成** |
| Qdrant 连接失败 | L3 RAG 返回空；生成降级为 L1+L2；`/vector` 页显示断连 |
| RAG rebuild 中途失败 | Celery retry 3 次；失败章进 Redis `failed_chapters`；前端按钮支持"仅重试失败章" |
| ask_user 超时 | 返回 `"(no reply — timed out)"`；pause `status=timeout`；不影响生成 |
| ContextPack 超 token budget | `_truncate_to_budget` 保留末尾；日志 `truncated=true` 并写进 `LLMCallLog` |
| 正文生成返回空 | 重试一次；仍空则 `LLMCallLog status=error` 并 SSE 推 error |
| Qdrant embedding 维度不匹配 | 启动时 `_ensure_collections` 检查 collection dim vs `router.embed("test")` 长度；不匹配 → log error + 提示手动迁移（不自动重建避免丢数据） |

## 8. 迁移步骤

Alembic 一次迁移：

1. `PromptAsset` 加列：`endpoint_id`, `model_name`, `temperature`, `max_tokens`, `category`, `order`, `always_enabled`, `name_en`, `description_en`。
2. 创建表 `llm_call_log`, `ask_user_pause`。
3. Python 数据迁移脚本：
   - 读取 `ModelConfig` 全部行。
   - 按 `task_type` 找对应 active `PromptAsset`；把 `endpoint_id/model_name/temperature/max_tokens` 拷过去。
   - 内置 `BUILTIN_PROMPTS`（9 条）中如未 seed 则 seed。
   - 为每条 builtin prompt 设置合理 `category` 和 `order`（见 5.1）。
4. `op.drop_table("model_config")`。
5. （可选）提供 `scripts/rebuild_all_rag.py` 作为命令行触发回溯。

## 10. 风险与开放问题

1. **ChapterGenerator 重写的回归风险**：`context_assembler.build_context_for_chapter` 被多处调用（`generate.py`, 可能还有其他批量生成路径）。实施阶段需扫一遍所有调用点，一次性切换。
2. **token 预算冲击**：ContextPack 默认 budget 8000，若 RAG 命中 5 条 each 200 字 ≈ 667 token，需确认不同端点模型的 context window 够用（Claude 4 / GPT-4o 都够；本地 Qwen 7B 可能紧）。实施时在 settings 里暴露 per-prompt token_budget 字段？暂不做，v0.5 统一 8000。
3. **LLMCallLog 膨胀**：每次生成/提取/评估都写一行，长篇会很快上万条。v0.5 不自动清理，前端提供"按项目清理"；生产阶段观察表体积再定策略。
4. **ask_user 场景覆盖面**：v0.5 不定义哪些 prompt 允许触发 `ask_user`，只提供基础设施；实际接入是 v0.6 的任务。
5. **Qdrant 维度不匹配**：若用户换了 embedding 模型，老数据失效。v0.5 只报错不自动重建；用户需在 `/vector` 页手动 delete collection → rebuild。

## 11. 验收标准

- [ ] `/prompts` 页能给每个 prompt 单独绑端点、改模型、调温度，`/settings` 页不再显示任务路由区块。
- [ ] 建新 prompt（如 `polish_dialogue`）并绑独立端点，保存后能在生成中被使用（通过 `/logs` 验证）。
- [ ] `/vector` 页能列出 3 个 collection、显示每个的 count/dim、预览最近 payload、支持检索测试、能删点。
- [ ] 点"重建 RAG"后能看到进度，完成后 `chapter_summaries` count 等于项目章节数。
- [ ] `/logs` 能打开任一条日志查看完整 messages、RAG 命中片段、响应、tokens、延迟。
- [ ] 生成一章后去 `/logs` 查该章节，system message 里应包含 `=== 细节参考 ===` 段（即 RAG 层生效）。
- [ ] workspace 模拟一次 `ask_user`，弹窗出现、答复后生成继续。
- [ ] Alembic 从 v0.4 库升级到 v0.5 无错，`ModelConfig` 表消失，所有旧路由转移到 PromptAsset。
- [ ] 所有测试通过。
